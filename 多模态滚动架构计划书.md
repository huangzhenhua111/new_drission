更新时间：2026-06-23

# 0.最终总目标
/home/huangzhenhua/workspace/new_drission/面试作业：Web 自动化脚本生成 Agent.pdf

## 1. 核心判断

之前的系统主要问题不是“容错不够多”，而是 Generation 一开始就把网页路径想死了。

对于新网站，文本 LLM 不应该提前生成一串具体按钮点击步骤，因为它没有真实看到页面结构。正确方式应该是：

1. 先把用户需求整理成全局任务状态 JSON；
2. 把任务拆成有顺序的 milestone，而不是拆成固定点击步骤；
3. 每一步都读取当前全局 JSON、DOM、截图、URL 和近期 trace；
4. 用多模态模型决定下一步最小动作；
5. 执行后更新全局 JSON；
6. 失败时 Debugger 根据 trace 找到错误点，从最近可信 URL 回退后继续 Generation。

一句话概括：

> Generation 不是提前写死完整网页流程，而是维护任务状态，边观察边决定下一步。

## 2. 顶层仍然对齐 PDF 三模块

虽然内部实现会更像“任务理解与状态管理 + 执行与感知 + 恢复纠错”，但对外仍然保持 PDF 要求的三个模块：

```text
Generation
  负责理解任务、维护全局 JSON、逐步生成下一步动作。

Resilience / 容错
  负责安全执行动作、处理登录/验证、等待、重试、selector fallback 和 trace 记录。

Debugger
  负责根据 trace、截图和全局 JSON 定位错误步骤，选择 rollback_url，生成修复提示并继续 Generation。
```

## 3. 第一版架构总流程

```text
用户需求
  ↓
Text LLM 生成 task_state.json
  ↓
进入 Generation 循环
  ↓
读取：
  - task_state.json
  - 当前 DOM candidates
  - 当前截图
  - 当前 URL/title/text
  - 最近 trace
  ↓
多模态模型输出下一步 tool_call JSON
  ↓
Resilience Executor 执行动作
  ↓
更新 trace 和 task_state.json
  ↓
验证当前动作 / 当前 milestone
  ↓
成功：进入下一轮或下一 milestone
失败：进入 Debugger
  ↓
Debugger 找 root cause step 和 rollback_url
  ↓
goto rollback_url
  ↓
带 repair_instruction / avoid 继续 Generation
```

### 4.1 必须保留的字段

第一版至少要保留：

- `original_user_request`
- `site.entry_url`
- `site.current_url`
- `site.last_stable_url`
- `resources`
- `goal.success_criteria`
- `milestones`
- `current_milestone_id`
- `rolling_context.previous_step`
- `rolling_context.current_page`
- `rolling_context.next_step`
- `stable_points`
- `recent_trace`
- `known_failures`
- `avoid`

这些字段共同回答几个问题：

- 用户到底想做什么？
- 资源文件在哪里？
- 当前做到哪个 milestone？
- 上一步在哪个页面做了什么，结果是否成功？
- 当前在哪个页面，页面上和当前目标相关的元素是什么？
- 下一步要推进哪个 milestone，具体 intent 和成功条件是什么？
- 哪些 milestone 已经有证据证明完成？
- 当前页面是从哪里来的？
- 最近做过什么？
- 哪些动作已经证明不要再试？
- 如果走错了，最近能回到哪个 URL？

### 4.2 rolling_context 是每轮决策的短期工作记忆

`task_state.json` 不能只是长期任务账本，还必须显式告诉模型“刚刚发生了什么、现在在哪、下一步想干什么”。

因此第一版必须保留：

```text
rolling_context.previous_step
  上一步在哪个页面做了什么，期望是什么，实际结果是什么。

rolling_context.current_page
  当前在哪个页面，页面大概是什么状态，哪些可见元素和当前目标有关。

rolling_context.next_step
  下一步要推进哪个 milestone，具体 intent 是什么，成功条件是什么，哪些事现在不要做。
```

每次调用多模态 Step Decider 之前，都必须把这三个上下文放在 prompt 的靠前位置。

如果缺少这三个字段，模型会变成“每一步重新看图猜谜”，容易出现看见 Open Sans 就点 Open Sans、看见 Export 就提前导出的情况。

更新规则：

1. 动作执行前，根据当前页面刷新 `current_page`；
2. 动作执行时，把当前目标写入 `next_step.intent` 和 `next_step.success_condition`；
3. 动作执行后，把动作、期望结果、实际结果写入 `previous_step`；
4. 如果当前 milestone 完成，推进 `current_milestone_id`，并重写 `next_step`；
5. 如果动作失败，把失败摘要写入 `known_failures` 和 `avoid`。

## 5. Generation 模块新职责

Generation 分成两个子职责。

### 5.1 Task Parser：生成全局任务 JSON

输入：

- 用户原始需求；
- 可选的网站 URL；
- 可选资源路径；
- 当前项目上下文。

输出：

- `task_state.json` 初版。

它只负责拆 milestone，不负责预测具体网站按钮。

错误示例：

```text
1. 点击 Create Project
2. 点击 Upload
3. 点击 Text
4. 点击 Export
```

正确示例：

```text
m1: 进入可编辑的视频工作区
m2: 上传 sample.mp4 并确认进入时间线
m3: 剪掉开头前 2 秒
m4: 将速度调整为 1.5 倍
m5: 添加标题文字
m6: 降低不透明度
m7: 点击导出
```

### 5.2 Step Decider：多模态滚动决策

每一轮输入：

- 精简后的 `task_state.json`；
- `rolling_context.previous_step`：上一步在哪个页面做了什么，实际结果是什么；
- `rolling_context.current_page`：当前在哪个页面，和当前目标相关的可见状态是什么；
- `rolling_context.next_step`：下一步要推进什么 intent，成功条件是什么，哪些事现在不要做；
- 当前 DOM candidates；
- 当前截图；
- 当前 URL/title/text；
- 最近 trace；
- 当前 milestone；
- `avoid` 和 `known_failures`。

Step Decider 的核心任务不是“重新规划整个任务”，而是只回答一个问题：

> 基于上一步结果、当前页面状态、下一步 intent，现在最小且最安全的一步动作是什么？

输出严格 JSON：

```json
{
  "action_type": "click",
  "target_candidate_id": "cand_42",
  "value": null,
  "path": null,
  "reason": "该按钮看起来是进入视频编辑器的入口",
  "confidence": 0.86,
  "expected_result": "进入编辑器工作区或出现上传入口"
}
```

允许的 `action_type` 第一版限定为：

- `click`
- `input`
- `upload`
- `select`
- `wait`
- `scroll`
- `hotkey`
- `finish`
- `ask_user`

原则：

- 优先选择 DOM `candidate_id`；
- 不让模型直接自由返回坐标；
- 只有没有可靠 DOM 映射时，才允许坐标 fallback；
- 每个动作必须给出 `expected_result`；
- 每个动作执行后必须写入 trace。

## 6. 第一版可以全走多模态

为了优先提高新网站成功率，第一版可以把所有下一步决策都交给多模态模型。

也就是策略上先等价于：

```text
risk_policy = always_ask_vision
```

但代码结构仍然保留风险分流接口：

```text
DOM candidates
  ↓
RiskGate
  ↓
低风险：未来可直接 DOM 执行
中高风险：问多模态模型
```

第一版可以先把所有动作都视为中高风险。后续提速时，再逐步开放低风险 DOM 直通。

## 7. Resilience / 容错模块职责

容错模块不是失败后才出现，而是每一步执行都要经过它。

第一版职责：

1. 校验 Generation 输出的 action JSON；
2. 确认 `target_candidate_id` 存在；
3. 确认目标 DOM 可见、可操作；
4. 执行 click/input/upload/select/wait/scroll/hotkey；
5. 处理 selector fallback；
6. 处理页面加载等待；
7. 检测登录、验证码、人机验证；
8. 需要用户登录时暂停自动执行；
9. 用户登录完成后继续 Generation 循环；
10. 记录 before/after URL、截图、DOM 摘要、动作结果；
11. 更新 `task_state.json` 的 `current_url`、`recent_trace`、`known_failures`。

登录/验证流程：

```text
检测到登录/验证
  ↓
暂停自动动作
  ↓
提示用户在当前 Chrome UI 完成
  ↓
轮询 URL/页面状态
  ↓
登录完成
  ↓
继续当前 milestone 的 Generation 循环
```

注意：登录不是任务失败，不应该直接结束任务。

## 8. Debugger 第一版：不做复杂 checkpoint，只做 URL 回退

第一版 Debugger 不恢复完整浏览器状态，也不做复杂 checkpoint。

它只做四件事：

1. 读取 `task_state.json`、trace 和截图；
2. 找出最可能的错误步骤；
3. 从 `stable_points` / `last_stable_url` 找出 `rollback_url`；
4. 生成 `repair_instruction` 和 `avoid`，然后回到该 URL 继续 Generation。

Debugger 输出示例：

```json
{
  "debug_result": "rollback_and_continue",
  "root_cause_step_id": "s8",
  "rollback_url": "https://example.com/projects/abc",
  "rollback_reason": "s7 后仍在项目编辑页，是最近可信状态",
  "repair_instruction": "不要再次点击字体下拉框；应双击画布中的文本框并输入标题文字",
  "update_task_state": {
    "current_milestone_id": "m5",
    "avoid": [
      "不要把字体选择框当成标题文本输入框"
    ]
  },
  "confidence": 0.82
}
```

执行方式：

```text
goto rollback_url
  ↓
observe 当前页面
  ↓
把 repair_instruction 和 avoid 写入 task_state.json
  ↓
继续当前 milestone 的 Generation 循环
```

### 8.1 rollback_url 的来源

优先级：

1. `stable_points` 中最近的可信 URL；
2. 当前 milestone 开始时的 URL；
3. `site.last_stable_url`；
4. `site.entry_url`；
5. 无法回退时停止并说明原因。

### 8.2 什么时候记录 stable point

满足以下任意条件时记录：

- 成功进入项目页 / 编辑器页；
- 某个 milestone 完成；
- 上传完成后页面仍可访问；
- URL 从首页变成具体项目 URL；
- 登录完成后的落点页面；
- action 后 postcondition 通过；
- Debugger 判断某个 URL 是可信恢复点。

不要每一步都记录 stable point，否则 Debugger 会很难判断真正可信的位置。

## 9. Trace 第一版需要记录什么

每一步至少记录：

```json
{
  "step_id": "s12",
  "milestone_id": "m5",
  "url_before": "...",
  "url_after": "...",
  "screenshot_before": "screenshots/s12_before.png",
  "screenshot_after": "screenshots/s12_after.png",
  "dom_before": "dom_snapshots/s12_before.json",
  "dom_after": "dom_snapshots/s12_after.json",
  "action": {
    "action_type": "input",
    "target_candidate_id": "cand_88",
    "value": "短视频测试"
  },
  "reason": "当前 milestone 需要添加标题文字",
  "expected_result": "画布中出现标题文字",
  "result": "failed",
  "failure": {
    "type": "postcondition_failed",
    "message": "画布中没有检测到可读标题文字"
  }
}
```

完整 trace 可以存文件，prompt 中只放最近几步摘要。

## 10. 后续提速策略

第一版先追求成功率，可以全走多模态。后续再逐步提速：

1. 对唯一且高置信 DOM candidate 直接执行；
2. 对已验证成功的 selector 建立缓存；
3. 对同一网站/同一页面模式复用动作模板；
4. generated script replay 阶段不再每步问模型；
5. 只有失败或页面变化时再调用多模态模型；
6. trace 和 DOM 默认保存压缩版，失败时再保存完整上下文。

## 11. 第一阶段实施任务

第一阶段只实现最小闭环，不做复杂 checkpoint。

### 11.1 新增核心文件

建议新增：

```text
app/generation/task_state.py
app/generation/multimodal_decider.py
app/recovery/url_rollback_debugger.py
```

### 11.2 改造 Generation

- Text LLM 生成 `task_state.json`；
- 不再把 planner 输出当作固定点击路线；
- 每轮根据当前页面和当前 milestone 调用多模态 decider；
- 多模态 decider 输出 tool_call JSON；
- 执行后更新 `task_state.json`。

### 11.3 改造 Runtime / 容错

- 所有动作统一经过 resilient executor；
- 每步记录 before/after trace；
- 登录/验证时暂停等待用户；
- 用户完成后继续 Generation，不结束任务；
- action 失败时把 failure context 交给 Debugger。

### 11.4 实现 URL Rollback Debugger

- 读取 `task_state.json` 和 trace；
- 找 root cause step；
- 选择 rollback_url；
- 生成 repair_instruction；
- 更新 `avoid`；
- 跳回 rollback_url；
- 继续 Generation。

## 12. 第一阶段验收标准

至少满足：

1. 能从用户自然语言需求生成 `task_state.json`；
2. `task_state.json` 中有 milestone、资源、当前状态、stable URL；
3. `task_state.json` 中显式包含 `rolling_context.previous_step/current_page/next_step`；
4. Generation 每步输入 DOM + 截图 + task_state + rolling_context；
5. 多模态模型输出结构化 tool_call JSON；
6. Runtime 优先用 DOM candidate 执行动作；
7. 每步留下 trace，并同步更新 rolling_context；
8. 登录/验证后不会直接结束任务；
9. 失败后 Debugger 能输出 root cause、rollback_url、repair_instruction；
10. 系统能跳回 rollback_url 继续当前 milestone；
11. 成功动作能被记录，为后续生成稳定脚本做准备。

## 13. 第一版暂不做

以下内容后置：

- 完整浏览器状态 checkpoint；
- 任意步骤的精确状态恢复；
- 大规模低风险 DOM 直通；
- 自动脚本 patch 闭环；
- 多网站模板库；
- 长期记忆压缩策略；
- 复杂拖拽/画布内部状态恢复。

这些不是不要，而是等最小闭环跑通后再做。

## 14. 当前建议下一步

1. `task_state.json` 数据结构；
